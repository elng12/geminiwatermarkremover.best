# PRD Review Evaluation（历史归档说明）

## 1. 状态说明

这份文档保留为**历史评审归档**，用于解释项目曾经考虑过哪些路线。

它不再代表当前执行方案。

当前项目已经明确切换到：

- 单首页 SEO 架构
- 纯浏览器图片处理
- 不接远程模型
- 不做云端兜底
- 不上传图片处理

如与当前方案冲突，以以下文档为准：

- `docs/gemini-watermark-remover-prd.md`
- `docs/ai-build-brief.md`
- 其对应的中文 / 英文施工文档

---

## 2. 这份历史评审里哪些观点仍然有效

以下判断仍然有价值：

- 首页应是工具优先，而不是长文优先
- `free`、`online` 这类小变体不应该拆很多薄页
- FAQ、隐私页、服务条款页必须补齐
- 不应承诺移除 SynthID、隐形水印或 provenance metadata
- 商标免责声明必须出现
- 移动端结果对比必须认真设计
- 示例图和 before / after 证明很关键

---

## 3. 这份历史评审里哪些假设已经过时

以下历史假设已不再适用：

- 需要选择远程处理引擎
- 需要决定自建模型还是第三方 API
- 需要设计云端处理状态、重试和删除策略
- 需要设置模型成本、每日额度和预算保护作为产品主线
- 需要规划多个支撑页来承接相似关键词

这些内容之所以过时，是因为当前已经确认：

> v1 只做浏览器内去除可见 Gemini sparkle 水印。

---

## 4. 当前正确的项目判断

当前版本最准确的结论是：

- 产品形态：**单首页工具站**
- 处理路线：**纯浏览器内处理**
- 去除目标：**可见 Gemini sparkle 水印**
- 非目标：**SynthID / invisible watermark / provenance metadata / 视频**

因此，当前最需要做的不是继续讨论模型，而是：

1. 做好首页转化与 SEO 文案
2. 实现浏览器内自动检测与去除
3. 做好手动选区重试
4. 准备真实 demo 图
5. 补齐法务页面与免责声明

---

## 5. 归档结论

这份评审在项目早期有帮助，但它建立在“可能走远程处理或多页面 SEO”的旧前提上。

现在这些前提已被放弃。

所以，这份文档的正确用途是：

- 作为历史背景保留
- 帮团队理解曾经为什么会讨论模型、限流、支撑页
- 不再作为当前开发、设计或 SEO 的直接执行依据
